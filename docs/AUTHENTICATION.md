# Authentication System

## Overview

AloMedia implements a modern, secure authentication system using httpOnly cookies for token storage. The system is built on a trust-based architecture where the backend maintains the source of truth for user sessions, and the frontend synchronizes its state accordingly.

## Architecture

### Session Management Strategy

Unlike traditional approaches that store authentication tokens in localStorage or sessionStorage, AloMedia uses **httpOnly cookies** as the primary authentication mechanism. This approach provides several security benefits:

- **XSS Protection**: Tokens stored in httpOnly cookies cannot be accessed by JavaScript, preventing theft through XSS attacks
- **CSRF Protection**: Cookies can be configured with SameSite policies to prevent unauthorized cross-site requests
- **Automatic Inclusion**: The browser automatically includes httpOnly cookies in HTTP requests without requiring manual handling

The token is generated by the backend upon successful login and stored in an httpOnly cookie. This means the frontend doesn't need to store the token anywhere—it's automatically sent with every subsequent request.

### Authentication Flow

The authentication system consists of four main phases:

#### Phase 1: Application Initialization
When the app loads, `AuthProvider` runs its `useEffect` hook which verifies the current session:

1. The `me()` endpoint is called with no parameters
2. Since the httpOnly cookie is automatically attached to this request, the backend can verify it
3. The backend responds with either user data (if authenticated) or an unauthenticated error
4. `isLoading` is set to `false` and the application renders

This ensures the UI always reflects the true authentication state, preventing brief flashes of incorrect content.

#### Phase 2: User Login
When a user submits their credentials on the login page:

1. `signIn()` sends email and password to `/auth/login`
2. Backend validates credentials and generates a token
3. Backend stores the token in an httpOnly cookie in the response
4. Frontend receives user data in the response body
5. Frontend calls `login(userData)` to update context state
6. Router redirects to `/dashboard`

The frontend never directly handles the token—it's automatically managed by the browser through the httpOnly cookie.

#### Phase 3: Authenticated Requests
Once authenticated, all subsequent requests include the token automatically:

1. Any fetch request made through the `http()` client automatically includes cookies
2. Backend middleware verifies the token in the cookie
3. If valid, the request proceeds; if expired or invalid, the backend returns 401 Unauthorized
4. Frontend error handling detects 401 errors and can trigger re-authentication

#### Phase 4: User Logout
When a user clicks logout:

1. `signout()` is called, making a POST request to `/auth/logout`
2. Backend invalidates the session and clears the httpOnly cookie
3. `AuthProvider` clears its local user state
4. Router redirects to `/auth/login`

## Key Components

### AuthProvider Context (`src/context/AuthProvider.tsx`)

The `AuthProvider` wraps the entire application and exposes:

**Context State:**
- `user`: The currently authenticated user object (or null)
- `isAuthenticated`: Boolean derived from `user !== null`
- `isLoading`: Boolean indicating whether authentication is being verified on app load

**Context Actions:**
- `login(user)`: Set the authenticated user after successful login
- `logout()`: Call the backend logout endpoint and clear local state

**Behavior:**
The provider automatically verifies the session on mount by calling the `me()` endpoint. This happens silently and ensures that:
- Users who were logged in previously are still authenticated if their session hasn't expired
- Users with invalid or expired tokens are treated as unauthenticated
- The UI reflects the correct authentication state before rendering

### Authentication Service (`src/services/authService.ts`)

Service functions that interact with authentication endpoints:

**`signIn(payload)`**: POST to `/auth/login` with email and password

**`signUp(payload)`**: POST to `/auth/register` with email, password, and other registration data

**`me()`**: GET from `/auth/me` to verify current session and get user data

**`signout()`**: POST to `/auth/logout` to invalidate the session

**`recoverRequest(payload)`**: POST to `/auth/recover/request` with email to request password recovery

**`validateRecoverToken(token)`**: GET to `/auth/recover/validate?token=` to check if a recovery token is valid

**`recoverReset(payload)`**: POST to `/auth/recover/reset` with token and new password to complete password recovery

### HTTP Client (`src/api/http.ts`)

A generic HTTP client that wraps the Fetch API:

```
http<T>(endpoint, options)
- Automatically includes credentials (cookies)
- Parses JSON responses by default
- Handles error responses consistently
- Provides type-safe responses
```

The client automatically attaches `credentials: 'include'` to all requests, ensuring httpOnly cookies are sent with every request.

## User Types (`src/types/userTypes.ts` and `src/types/authTypes.ts`)

### User Type
Represents an authenticated user with properties like:
- `id`: Unique user identifier
- `email`: User's email address
- `name`: User's display name
- Additional profile fields as needed

### Auth Types
Request/response types for authentication endpoints:
- `LoginPayload`: Email and password
- `RegisterPayload`: Registration form data
- `AuthResponse`: User data returned on login/register
- `MeResponse`: User data and authentication status
- `RecoverRequestPayload`: Email for recovery
- `RecoverResetPayload`: Token and new password

## Password Recovery

The password recovery system provides a secure way for users to reset forgotten passwords:

### Recovery Request Phase
1. User navigates to `/auth/recover/request`
2. Enters their email address
3. Backend generates a time-limited recovery token and sends it via email
4. User clicks the link in the email, which routes to `/auth/recover?token=xyz`

### Token Validation Phase
1. Frontend validates the token is present in the URL
2. Can optionally call `validateRecoverToken()` to ensure it's still valid
3. Shows password reset form only if token is valid

### Password Reset Phase
1. User enters new password and confirmation
2. Frontend submits `recoverReset()` with token and new password
3. Backend validates the token hasn't expired and password meets requirements
4. Backend updates the password and clears the recovery token
5. User is redirected to login page

## Security Best Practices Implemented

### CORS Headers
The application configures CORS headers in `vite.config.ts` to work with SharedArrayBuffer for FFmpeg's multi-threading, while maintaining security.

### HTTP-Only Cookies
By using httpOnly cookies instead of localStorage:
- XSS attacks cannot steal authentication tokens
- The browser prevents JavaScript access to the token
- Only HTTP(S) requests include the token

### Time-Limited Authorization
- Session tokens have expiration times
- Recovery tokens have very short expiration windows
- Expired tokens are rejected by the backend

### Error Handling
The HTTP client catches auth errors and provides appropriate feedback without exposing sensitive implementation details.

## Integration with Protected Routes

The authentication state from `AuthProvider` is used by route guards:

**PrivateRoute**: Checks if `isAuthenticated` is true. If false, redirects to login.

**PublicRoute**: Checks if `isAuthenticated` is true. If true, redirects away from auth pages.

This creates a secure routing system where:
- Unauthenticated users cannot access private content
- Authenticated users cannot see login/register pages
- Some routes (like `/editor`) are currently unprotected for development but can be moved under `PrivateRoute` for production

## Session Verification Flow

When users refresh the page or return to the app after it's been idle:

1. App initializes and mounts `AuthProvider`
2. AuthProvider's `useEffect` calls the `me()` endpoint
3. Backend checks if the httpOnly cookie contains a valid session
4. If valid, backend returns user data; if invalid, returns unauthenticated response
5. AuthProvider updates its state based on the response
6. React Router guards ensure correct routes are shown based on new state

This automatic verification ensures that:
- Users don't remain authenticated if their session expires
- Users who logout in another tab/window are logged out in this one
- The UI always reflects the true server state

## Error Handling

Common authentication errors and how they're handled:

- **401 Unauthorized**: User's session is invalid or expired. Redirect to login.
- **400 Bad Request**: Validation error in request data. Show form errors.
- **409 Duplicate**: Email already registered. Suggest login or password recovery.
- **404 Not Found**: Email not found for recovery request. May show generic message for security.
- **403 Forbidden**: Insufficient permissions. Show error message.
